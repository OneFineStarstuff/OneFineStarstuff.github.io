Phase 1: Years 1-2 -- Tier 0/1 Hardening + AI API Gateway Foundation
Objective: Harden privileged tiers to ESAE standard, deploy isolated AI API gateways at tier boundaries, establish baseline telemetry.
Y1-H1 Tier 0 Hardening
Months 1-6
Strategic Milestones
  • Complete Tier 0 isolation: Migrate all domain controllers to dedicated hardware (no hypervisor co-tenancy). Deploy hardened Server Core 2025 images with AppLocker SRP + WDAC.

  • AI readiness assessment: Catalog all existing AI/ML workloads (data pipelines, LLM endpoints, copilot integrations). Map each to current tier crossing points. Produce AI Agent Threat Model v1.

Friction-Resolution Pattern: Tier-Scoped API Gateway with OPA Sidecar
Friction: AI agents need to query Tier 1 service data (server configs, patch status) but tier model prohibits Tier 2 identities from authenticating to Tier 1 resources.
Resolution: The API gateway acts as a tier-translation proxy. AI agents authenticate to the gateway with Tier 2-scoped OAuth tokens. The gateway, running as a gMSA identity within Tier 1, forwards sanitized read-only queries to Tier 1 APIs. The OPA sidecar enforces: (a) only GET operations, (b) no credential/secret fields in responses, (c) data classification ≤ "Internal". The AI agent never holds a Tier 1 credential -- the gateway is the tier boundary enforcer.

Friction: AI agents need real-time T0 security signals but T0 must have zero inbound trust.

Friction-Resolution Pattern: Dual-Authorization Write Gate
Friction: AI agents need to execute remediations (e.g., isolate compromised endpoint) but autonomous write access to enterprise systems violates the least-privilege/human-oversight principle.
Resolution: The gateway implements a propose-approve-execute pattern. The AI agent submits a structured remediation request (JSON schema-validated). The gateway holds the request and pushes a ServiceNow approval record. A human SOC analyst reviews and approves/rejects within SLA (≤ 15 min). Only upon approval does the gateway execute the write. All writes are Tier 2 only. Every action is immutably logged. Pre-change snapshots enable auto-rollback. This preserves AI velocity while maintaining human accountability.
Phase 2: Year 3 -- Zero Trust Bridging with ZTNA, OIDC Federation, Ephemeral Access
Objective: Replace static tier boundaries with continuous-verification zero trust policy enforcement. AI agents become first-class ZTNA subjects with ephemeral, scope-bound identities.
Y3-H1 ZTNA Policy Engine + OIDC Federation
Months 25-30
  • ZTNA policy engine deployment: Deploy centralized Policy Decision Point (PDP) -- e.g., Zscaler Private Access or Cloudflare Access -- as the universal access broker. All cross-tier access (human and AI) flows through the PDP. Policy evaluation inputs: identity, device/agent posture, resource sensitivity, time, location, behavioral risk score.
  • OIDC federation for AI agents: Register each AI agent class as an OIDC Relying Party (RP) against the enterprise IdP (Entra ID). Agents authenticate via OIDC Authorization Code Flow with PKCE (RFC 7636) -- not client credentials. Tokens include custom claims: {"tier_scope": "2", "action_class": "read", "risk_ceiling": "medium"}.
  • Conditional Access policies for AI: Extend Entra ID Conditional Access to AI agent identities: require compliant agent posture (signed binary, valid attestation), evaluate real-time risk score, enforce step-up for Tier 1 read access (additional claim required from PDP).
  • AI-to-AI identity mesh: Agents communicating with each other authenticate via SPIFFE/SPIRE (Secure Production Identity Framework for Everyone). SPIFFE IDs: spiffe://corp.internal/ai/agent/{class}/{instance}. Workload attestation via K8s pod identity.

Friction-Resolution Pattern: Continuous-Verification Identity Bridge
Friction: Static tier boundaries are binary (in/out). AI agents need graduated, context-dependent access that can be revoked in real-time.
Resolution: Replace binary tier membership with continuous-verification ZTNA. Every AI agent request is evaluated against real-time policy: posture check (is the agent binary signed and unmodified?), behavioral risk score (is this request pattern anomalous?), resource sensitivity (what tier does the target belong to?), and temporal scope (is this within the approved maintenance window?). No request is pre-approved; every request is individually evaluated. Entra ID CAE enables sub-minute revocation -- a compromised agent's tokens become invalid within 60 seconds.
Y3-H2 Ephemeral Access + AI T1 Read Path
Months 31-36
  • Ephemeral access model for Tier 1 AI reads: AI agents can now request Tier 1 read access via the ZTNA PDP. Access is ephemeral: scoped to a single API call, bound to a specific resource, and auto-expires. No persistent Tier 1 sessions. Implementation: ZTNA PDP issues a one-time-use access proxy token that the PEP validates and consumes (single-use JWT with jti claim, RFC 7519 §4.1.7).

  • Phase 2 audit: Red team exercise: attempt T0 access via AI agent compromise (supply chain, credential theft, prompt injection). Validate that ZTNA + tier controls prevent lateral movement.

Friction-Resolution Pattern: Ephemeral Single-Use Token with Behavioral Gating
Friction: AI agents need Tier 1 data (server configs, vulnerability scans) to make intelligent decisions, but persistent T1 access is a privilege escalation risk.
Resolution: Ephemeral, single-use access tokens that are: (a) issued per-request by the ZTNA PDP after continuous policy evaluation, (b) valid for ≤ 5 minutes and consumed on first use (JTI tracking prevents replay), (c) scoped to a single resource and operation, (d) gated by the agent's real-time behavioral risk score. If the agent's behavior deviates > 3-sigma from baseline, the PDP auto-denies and SPIRE force-rotates the agent's identity -- effective kill switch in < 60 sec.
Phase 3: Years 4-5 -- Behavioral API Sidecars, Autonomic Remediation, Quantum-Resistant Interplay
Objective: Enable AI agents to autonomously remediate cross-tier security incidents with behavioral safety nets and post-quantum cryptographic foundations.
Y4-H1 Behavioral API Sidecars
Months 37-42
  • Behavioral API sidecar deployment: Every AI agent pod runs a co-located behavioral sidecar (Envoy-based, custom WASM filter). The sidecar intercepts all outbound API calls in real-time and evaluates them against: (a) the agent's behavioral baseline, (b) the current policy set, (c) semantic analysis of the request payload (detecting anomalous patterns like bulk data exfiltration or privilege escalation attempts).
  • Sidecar circuit-breaker pattern: If the sidecar detects anomalous behavior (Z-score > 2.5 on any dimension), it trips the circuit breaker: blocks the outbound call, quarantines the agent pod (network policy update via Cilium), generates a SOC alert, and preserves forensic state (memory dump, request log snapshot).
  • AI write access extended to Tier 1 (scoped): With sidecar enforcement in place, AI agents can now execute pre-approved Tier 1 write operations (e.g., quarantine a compromised server, rotate a service account password). Still gated by: sidecar approval + ZTNA PDP + provenance logging. Human approval removed for pre-approved playbook actions only (enumerated allow-list: ≤ 20 operations).
  • Tier 0 write: still prohibited. No automated AI write access to Tier 0 under any circumstance. T0 remains human-only with PAW + JIT.

Friction-Resolution Pattern: Behavioral Sidecar as Inline Safety Net
Friction: Enabling AI autonomous remediation at Tier 1 risks runaway actions (e.g., an agent in a feedback loop quarantining healthy servers).
Resolution: The behavioral sidecar is an independent safety net co-located with every agent. It has its own behavioral model, independent of the agent's logic. Even if the agent is compromised or malfunctioning, the sidecar can independently detect anomalous patterns and trip the circuit breaker. The sidecar is immutable (read-only filesystem, signed binary verified at pod startup via Sigstore/Cosign). The agent cannot disable, modify, or bypass the sidecar -- it's enforced at the Kubernetes network policy level (Cilium: all egress must transit sidecar).
Y4-H2 Autonomic Remediation Engine
Months 43-48
  • Autonomic remediation engine v1: Deploy a centralized remediation orchestrator that coordinates multi-agent, multi-tier response sequences. Example: "Compromised T1 server detected -> AI Agent 1 quarantines server (T1 write) -> AI Agent 2 rotates affected credentials (T1 write) -> AI Agent 3 scans lateral movement indicators (T2 read) -> AI Agent 4 updates firewall rules (T1 write)." Entire sequence executes in < 3 min, with sidecar enforcement on every step.
  • Playbook-as-code: All remediation sequences defined in OPA Rego + CUE (machine-verifiable). Playbooks version-controlled in Git, code-reviewed by security architects, signed with Sigstore. Agents can only execute signed playbooks -- no ad-hoc actions.
  • Blast radius controls: Every playbook includes a max_blast_radius parameter: maximum number of resources affected in a single execution. Default: 5. Any action exceeding blast radius triggers human escalation regardless of playbook approval status.
  • Cross-tier incident correlation: AI agents correlate signals across T0 (telemetry lake), T1 (direct read), and T2 (direct read/write) to build unified incident timelines. Reduces mean-time-to-detect (MTTD) and mean-time-to-respond (MTTR).

Friction-Resolution Pattern: Signed Playbook-as-Code with Blast Radius Limits
Friction: Multi-step, multi-tier automated remediation could cause cascading failures if a playbook is flawed or an agent is compromised.
Resolution: Three independent safety layers: (1) Playbook signing -- agents can only execute code-reviewed, Sigstore-signed playbooks from the Git repository; no ad-hoc actions. (2) Blast radius limits -- every playbook declares the maximum number of resources it can affect; exceeding the limit triggers mandatory human escalation. (3) Sidecar enforcement -- each individual API call within the playbook is still evaluated by the behavioral sidecar; even a valid playbook is halted if its runtime behavior deviates from the expected pattern.

Friction-Resolution Pattern: Hybrid PQC Transition with Dual-Signing
Friction: PQC migration risks breaking existing Tier 0/1 services that rely on classical cryptography. A hard cutover is unacceptable for 99.9% uptime SLAs.
Resolution: Hybrid mode everywhere. Every certificate, token, and key exchange uses both classical and PQC algorithms simultaneously. Validators accept either. Over time (Y5-H2), classical algorithms are deprecated -- but the hybrid period ensures zero downtime during transition. The PQC root CA is cross-signed by the existing ECDSA root, establishing a chain-of-trust bridge. Agent sidecars transparently negotiate the strongest available algorithm per connection.
